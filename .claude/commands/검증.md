# /검증 — 프레임워크 또는 구현 검증

> **메타 검증 커맨드** — 4단계 게이트 면제.

## $ARGUMENTS

검증 대상 (선택): `프레임워크 [{카테고리}]` 또는 `구현 {LINEAR-ID}`. 미지정 시 `프레임워크` 전체 실행.

---

## 분기

| $ARGUMENTS | 실행 |
|-----------|------|
| `프레임워크` (또는 미지정 시 기본값) | 아래 §프레임워크 검증 — 전체 카테고리 실행 |
| `프레임워크 {카테고리}` | 해당 카테고리만 실행 (구조/참조/게이트/SSOT/컨텍스트/커맨드/스펙) |
| `구현 {LINEAR-ID}` | verify Skill 호출 — 구현 완전성 검증 |

---

## 프레임워크 검증

프레임워크 파일의 구조적 무결성을 25개 항목, 7개 카테고리로 검증한다.

### A. 구조 정합성

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| A-1 | 필수 파일 존재 | CLAUDE.md, `.claude/rules/pipeline.md`, `.claude/rules/docs-writing.md`, 각 스킬의 `SKILL.md`, 각 커맨드 `.md`. 프로젝트별 Rules(coding.md 등)는 CLAUDE.md `Project Rules`에 등록된 경우에만 검증 |
| A-2 | CLAUDE.md 줄 수 | `Framework: Forbidden` 기준 준수 (현재 200줄 제한) |
| A-3 | frontmatter paths 정합 | Rules 파일의 `paths:` 필드 vs CLAUDE.md `Framework: Rules Delegation` 테이블 일치 |
| A-4 | MCP 도구명 일관성 | 모든 스킬의 `## Linear MCP` 섹션에서 동일 도구명 사용 (`save_issue`, `get_issue`, `save_comment`, `list_issues`, `list_teams`, `list_issue_labels`, `list_projects`, `create_document`) |
| A-5 | 스킬 필수 섹션 완전성 | 모든 SKILL.md에 `## Trigger`, `## Input`, `## Process`, `## Output`, `## OMC 에이전트 연동`, `## Linear MCP` 6개 섹션(또는 명시적 변형) 존재 |
| A-6 | 파일 수 ↔ 선언 정합 | CLAUDE.md "커맨드 N개" 선언 vs 실제 `.claude/commands/*.md` 수. setup.md 선언도 교차 검증. 스킬도 동일 |

### B. 참조·링크 유효성

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| B-1 | Key References 경로 | CLAUDE.md `Key References` 섹션의 모든 경로가 실제 파일로 존재 |
| B-2 | 커맨드→스킬 매핑 | 각 커맨드가 참조하는 스킬 경로가 실제 존재 |
| B-3 | 템플릿 참조 유효성 | 스킬 내 `[...](templates/xxx.md)` 상대경로의 실제 파일 존재 확인 |
| B-4 | 스킬 간 상호참조 | `[...](../skill-name/SKILL.md)` 패턴의 링크 대상 파일 존재 확인 |
| B-5 | See Also 링크 | 커맨드의 `## See Also` 섹션 내 참조가 실제 존재하는 커맨드인지 확인 |
| B-6 | frontmatter ↔ 템플릿 정합 | docs-writing.md §1이 정의한 문서 유형별 필수 frontmatter 속성이 해당 템플릿에 포함되는지 확인 |

### C. 게이트·파이프라인

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| C-1 | 게이트 라벨 일관성 | 스킬 Process 테이블의 (G1)~(G4) 라벨이 pipeline.md §2 게이트 정의와 일치 |
| C-2 | 상태 전이명 유효성 | 모든 스킬/라우팅 테이블에서 참조하는 상태명이 pipeline.md §1-2의 7개 상태(`Backlog`, `Todo`, `Planning`, `In Progress`, `In Review`, `Done`, `Canceled`) 중 하나인지 확인 |
| C-3 | 라우팅 테이블 완전성 | routing-tables.md의 3개 타입(feature/improvement/bug) × 관련 상태 조합이 빠짐없이 정의 |
| C-4 | 확장 게이트 패턴 | spec의 G5/G3a, cleanup의 게이트 면제 등 비표준 게이트가 스킬 내 명시적으로 선언되어 있는지 확인 |
| C-5 | G3 이중기록 준수 | 게이트 면제가 아닌 모든 스킬의 G3 단계에서 Git 쓰기 행위 + Linear MCP 행위 양쪽 모두 명시 |

### D. SSOT 준수

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| D-1 | 3영역 SSOT 위반 스캔 | 기존 `docs/issue/*/` 내 `_index.md` frontmatter에 금지 필드(`status`, `tags`, `milestone`, `related`) 없는지 확인 |
| D-2 | _index.md 섹션 소유권 | 각 스킬의 Output 테이블에서 선언한 _index.md 갱신 섹션이 다른 스킬의 선언과 중복되지 않는지 확인 |

### E. 컨텍스트 최적화

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| E-1 | MCP 도구 ↔ 서버 정합 | 스킬의 Linear MCP 섹션에서 참조하는 도구명이 Linear MCP 서버의 실제 도구 목록에 존재하는지 확인 |
| E-2 | 컨텍스트 로드 규칙 | 각 스킬의 Input 테이블이 context.md §2 로드 매트릭스의 해당 단계 로드 범위를 초과하지 않는지 확인 |

### F. 커맨드 완전성

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| F-1 | 자체 포함 커맨드 구조 | 스킬 파일 없는 커맨드(`/커밋`, `/병합`, `/현황`, `/릴리스`)가 자체적으로 완전한 프로세스 정의(단계 테이블)를 포함하는지 확인 |
| F-2 | 파일 수 ↔ setup.md 정합 | setup.md §7 커맨드 요약 테이블의 항목 수와 실제 파일 수 일치 확인 |

### G. Spec 정합성

| ID | 검증 항목 | 세부 기준 |
|----|----------|---------|
| G-1 | Spec 템플릿 완전성 | spec 스킬이 참조하는 6개 템플릿 파일 모두 존재 + 각 템플릿에 frontmatter 블록 포함 |
| G-2 | Spec 글로벌 인덱스 정합 | `docs/spec/_index.md` 테이블의 항목과 실제 `docs/spec/*/` 디렉토리 일치 |

## 실행 단계

| 단계 | 행위 |
|------|------|
| 1 | `$ARGUMENTS`에서 카테고리 파싱 (미지정 시 전체 A~G) |
| 2 | CLAUDE.md 읽기 — Key References, Rules Delegation, 커맨드/스킬 수 선언 추출 |
| 3 | 대상 카테고리의 검증 항목 순차 실행 |
| 4 | 카테고리별 PASS/FAIL 집계 |
| 5 | 검증 결과 테이블 출력 (카테고리별 그룹화) |
