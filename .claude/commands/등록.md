# /등록 — Issue 등록

새 Issue를 Linear에 등록하고 Git 문서 폴더를 생성한다. **단건** 및 **배치(일괄)** 등록을 모두 지원한다.

## $ARGUMENTS

자유 형식 텍스트. 입력 길이와 내용에 따라 자동으로 모드를 판별한다.

- **단건 모드**: type 힌트 + 짧은 제목/설명. 예: `feature 사용자 인증 기능`, `bug 로그인 실패`
- **배치 모드**: 긴 텍스트(spec 문서, 보고서, 기획서, 자유 프롬프트 등). AI가 이슈 후보를 추출한다
- **미지정**: `AskUserQuestion`으로 입력 요청

### --from-spec 옵션 (선택)

| 항목 | 설명 |
|------|------|
| 형식 | `--from-spec {spec-name}` (kebab-case, `docs/spec/` 하위 디렉토리명) |
| 동작 | gen-hub Skill의 배치 모드 트리거. spec의 FR을 기반으로 Issue 배치 생성 |
| 예시 | `/등록 --from-spec combat-system` |

---

## 모드 판별

| 조건 | 모드 |
|------|------|
| 미지정 또는 `type + 한 줄 제목` 패턴 | **단건** |
| 여러 줄 텍스트, 문서 경로, 또는 복수 기능/버그 서술 | **배치** |

> 판별이 모호할 경우 `AskUserQuestion`으로 "단건 등록 / 배치 추출" 중 선택 요청

---

## 실행: 단건 모드

gen-hub Skill을 1회 호출한다.

| 단계 | 행위 |
|------|------|
| 1 | `$ARGUMENTS` 미지정 시 `AskUserQuestion`으로 type + 제목/설명 입력 요청 |
| 2 | `$ARGUMENTS`에서 type + 제목/설명 파싱 |
| 3 | gen-hub Skill 호출 — Linear Issue 생성 + `docs/issue/{LINEAR-ID}/_index.md` 생성 |
| 4 | 생성된 Issue 정보 출력 (LINEAR-ID, URL, type) |

---

## 실행: 배치 모드

입력 텍스트에서 이슈 후보를 추출하고, 사용자 확인 후 일괄 등록한다.

| 단계 | 행위 |
|------|------|
| 1 | **Linear 컨텍스트 조회**: `list_issue_labels` + `list_projects`를 **병렬** 호출하여 기존 Label 목록과 Project 목록을 1회 캐싱 |
| 2 | **텍스트 분석**: 입력 텍스트(또는 파일 경로 지정 시 해당 파일 내용)를 분석하여 이슈 후보 추출 |
| 3 | **후보 목록 제시**: 테이블 형태로 사용자에게 제시. 각 항목에 AI 추천 type, 제목, 설명, priority, **Labels, Project, 관계** 포함. 추천은 1단계에서 조회한 기존 항목 기반 |
| 4 | **사용자 확인**: `AskUserQuestion`으로 수정/삭제/추가/관계 변경 요청. 확인 시 다음 단계 진행 |
| 5 | **일괄 생성**: 의존 관계 순서대로 gen-hub Skill을 순차 호출. 선행 이슈 먼저 생성 → ID 획득 후 후속 이슈에 관계 설정. (1단계 캐싱 데이터를 전달하여 gen-hub 내부에서 재조회하지 않음) |
| 6 | **결과 요약**: 생성된 전체 Issue 목록 출력 (LINEAR-ID, URL, type, labels, project, 관계) |

### Linear 컨텍스트 조회 (1단계)

| MCP 도구 | 용도 |
|---------|------|
| `list_issue_labels` | 워크스페이스/팀의 기존 Issue Label 목록 조회 |
| `list_projects` | 활성 Project 목록 조회 (마일스톤 매핑용) |

> 조회 결과는 세션 내 캐싱. 배치 전체에서 재사용하며 매 건마다 재조회하지 않는다.

### 후보 목록 제시 형식

```
| # | type | 제목 | 설명 (요약) | priority | labels | project | 관계 |
|---|------|------|-------------|----------|--------|---------|------|
| 1 | feature | 인증 시스템 | ... | High | auth, backend | Sprint 12 | — |
| 2 | feature | 권한 관리 | ... | Medium | auth | Sprint 12 | blockedBy: #1 |
| 3 | improvement | 로그인 UX | ... | Low | ui | Sprint 12 | relatedTo: #1, #2 |
```

> - Labels/Project는 기존 Linear 항목에서 매칭. 새 라벨이 필요하면 `[new]` 표기
> - 사용자는 번호로 수정/삭제 지시 가능. 예: "2번 삭제", "3번 type을 feature로 변경", "1번 labels에 api 추가"

### 이슈 관계 설정

배치 내 이슈 간, 또는 기존 Linear 이슈와의 관계를 설정할 수 있다.

| 관계 유형 | 문법 | 설명 |
|-----------|------|------|
| `blockedBy` | `blockedBy: #2` 또는 `blockedBy: PRJ-42` | 선행 작업 (이 이슈가 차단됨) |
| `blocks` | `blocks: #3` | 후속 차단 (이 이슈가 차단함) |
| `relatedTo` | `relatedTo: #1, PRJ-42` | 관련 이슈 연결 |
| `parentId` | `parent: #1` 또는 `parent: PRJ-42` | 상위 이슈 (sub-issue) |

> - `#N`은 배치 내 번호 참조. 생성 후 실제 Linear ID로 치환
> - `PRJ-N`은 기존 Linear 이슈 직접 참조
> - `blockedBy`/`blocks` 관계가 있으면 선행 이슈를 먼저 생성하도록 순서 자동 정렬

### 제약

- 한 번에 최대 **10건**까지 등록. 초과 시 분할 안내
- gen-hub Skill 자체는 변경 없이 단건 호출 반복
- 순환 의존(A→B→A) 감지 시 사용자에게 경고 후 관계 수정 요청
