# /릴리스 — 프레임워크 템플릿 릴리스

> **메타 커맨드** — 4단계 게이트 면제. 프레임워크 템플릿 레포 전용.

## $ARGUMENTS

버전 (선택): `v1.1.0` 등 SemVer 형식. 미지정 시 사용자에게 major/minor/patch 선택 요청.

---

## 실행 흐름

### Step 0: 템플릿 레포 확인

`git remote get-url origin`으로 remote URL을 확인한다.
URL에 `claude-code-linear-template`이 포함되지 않으면 아래 메시지를 출력하고 **중단**한다:

> "이 커맨드는 프레임워크 템플릿 레포 전용입니다. 파생 프로젝트에서는 사용할 수 없습니다."

### Step 1: 버전 결정

- `$ARGUMENTS`에 버전이 지정되면 그대로 사용
- 미지정 시:
  1. `git tag --sort=-v:refname | head -1`로 최신 태그 확인
  2. `AskUserQuestion`으로 bump 유형 선택: `major` / `minor` / `patch`
  3. 최신 태그에서 자동 계산 (예: v1.0.0 + patch → v1.0.1)
  4. 태그가 없으면 `v1.0.0` 제안

### Step 2: 프레임워크 검증

`/검증 프레임워크` 실행.

- 전체 PASS: Step 3으로 진행
- 1건이라도 FAIL: FAIL 항목 안내 후 **중단**

### Step 3: 버전 반영

1. `README.md`의 `> 버전: v{OLD}` 패턴을 `> 버전: v{NEW}`로 교체
   - 패턴 매칭 실패 시 `AskUserQuestion`으로 수동 확인
2. `README.md` 버전 이력 테이블에 새 행 추가: `| v{NEW} | {YYYY-MM-DD} | {AskUserQuestion으로 주요 변경 입력} |`

### Step 4: 커밋

```
git add README.md
git commit -m "chore(release): v{VERSION}"
```

### Step 5: 태그 + Push

```
git pull --rebase origin main
git tag -a v{VERSION} -m "v{VERSION}"
git push origin main --tags
```

- `git pull --rebase` 실패 시: 충돌 안내 + 수동 해결 요청 후 중단
- `git push` 실패 시 (branch protection 등): 에러 안내 + "수동으로 PR을 생성하세요" 가이드

→ push 성공 시 GitHub Actions(`release.yml`)가 자동으로 Release + clean Asset 생성.

### Step 6: 완료 안내

```
릴리스 v{VERSION} 완료.
- GitHub Release: https://github.com/{owner}/{repo}/releases/tag/v{VERSION}
- GitHub Actions에서 clean 아카이브 Asset이 자동 생성됩니다.
```

---

## 첫 릴리스 특수 처리

태그가 0개인 경우(첫 릴리스):
- Step 5에서 `--generate-notes` 대신 `--notes`로 수동 초회 노트를 release.yml이 처리
- 단, 현재 release.yml은 항상 `--generate-notes` 사용. 첫 릴리스 시 전체 커밋 히스토리가 포함될 수 있음

## SemVer 기준

| Bump | 기준 | 예시 |
|------|------|------|
| major | 호환 불가능한 구조 변경 | 폴더 구조, 커맨드 이름 변경 |
| minor | 새 스킬/커맨드 추가 (하위 호환) | 새 type, 가이드 추가 |
| patch | 버그 수정, 문구 개선 | 오타, 참조 경로 수정 |
