# /점검 — 수동 테스트 결과 분류 및 라우팅

수동 테스트 결과를 8유형으로 분류하고 적절한 처리 경로로 라우팅한다.

## $ARGUMENTS

테스트 결과 텍스트 (자유 형식). 미지정 시 `AskUserQuestion`으로 사용자에게 입력 요청.

---

## 후처리 모드

분류(G0~G3) 완료 후 3-Mode 후처리(인터뷰/자동계획/스킵)를 선택한다.

> 모드 선택 테이블 및 상세: [triage SKILL.md](../skills/triage/SKILL.md) §후처리 파이프라인

---

## 독립 사용 시나리오

파이프라인 외부에서도 `/점검`을 독립적으로 사용할 수 있다.

| 시나리오 | 설명 |
|---------|------|
| **파이프라인 내** | implement 완료 후 수동 테스트 결과를 전달. Linear ID + plan.md 활용하여 in-scope/out-of-scope 판단. 후처리 모드 3가지 선택 가능 |
| **파이프라인 외 독립** | Linear ID 없이도 호출 가능. plan.md 없으므로 scope 판단은 사용자 컨텍스트 기반. **후처리는 Mode 3(스킵)만 허용** |

> Linear ID 없이 호출 시 triage는 독립 호출 모드로 동작 — plan.md 없이 8유형 분류 수행. 후처리는 Mode 3(스킵)만 허용.

---

## 실행

<!-- 참조 문서용 — 실행 경로가 아닌 스킬 참조 -->
triage Skill을 호출한다.

| 단계 | 행위 |
|------|------|
| 1 | `$ARGUMENTS` 미지정 시 `AskUserQuestion`으로 테스트 결과 입력 요청 |
| 2 | `$ARGUMENTS`에서 테스트 결과 텍스트 파싱 |
| 3 | 활성 Issue의 Linear ID 확인 (없으면 독립 호출 모드) |
| 4 | triage Skill 호출 — 9유형 분류 → G2 승인 → 후처리 모드 선택 → 실행 |

> triage Skill 상세: [.claude/skills/triage/SKILL.md](../skills/triage/SKILL.md)
